# Ingest

The ingest pipeline fetches vulnerability data from multiple vendor sources and normalizes it into the unified [Limoza Schema](schema.md).

## Pipeline Architecture

| Stage | Description |
|-------|-------------|
| **Sync** | Fetch raw data from vendor APIs/feeds and store locally |
| **Transform** | `parse()` + `transform()` per vendor module → `LimozaRecord` |
| **Upsert** | `upsert_lve_record()` in `db.py` writes to PostgreSQL |

**Sources and their sync targets:** each source has its own sync target (e.g.
`sync redhat`, `sync nvd`, `sync ubuntu`). Run `sync all` to fetch everything.

| Vendor | Sync command | Format |
|--------|-------------|--------|
| NVD / MITRE | `sync nvd` | NVD JSON 2.0 |
| Red Hat | `sync redhat` | CSAF 2.0 (VEX + advisories) |
| SUSE | `sync suse` | CSAF 2.0 (VEX + advisories) |
| Ubuntu / Debian | `sync ubuntu`, `sync debian` | OpenVEX / DSA feeds |
| AlmaLinux / Rocky / Oracle | `sync almalinux`, `sync rocky`, `sync oracle` | errata / OVAL |
| Microsoft MSRC | `sync microsoft` | CVRF v3.0 JSON |
| GHSA / OSV | `sync ghsa`, `sync osv` | OSV JSON |
| ExploitDB / Metasploit / Nuclei / PoC-in-GitHub | `sync exploitdb`, … | CSV / git / JSON |
| EPSS / CISA KEV / CISA SSVC / BSI | `sync epss`, `sync cisa_kev`, … | CSV / JSON |

See the [Ingest CLI](../running/cli.md) reference and the [data source pages](../datasources/index.md) for the full list.

## Vendor Module Structure

Each vendor module implements two functions:

| Function | Description |
|----------|-------------|
| `parse(raw: bytes) → dict` | Deserialize raw source data |
| `transform(doc: dict) → list[dict]` | Map source fields to Limoza records |

The `upsert_lve_record()` in `db.py` handles all database writes — vendor modules never touch the database directly.

## Import Order

Vendors can be imported in **any order**. The schema is order-independent.

Only NVD writes CVE spine fields (`cve_id`, `status`, `published`, `updated`, `epss`, `kev`, `ssvc`). All other vendors use `COALESCE(EXCLUDED.x, lve_cve.x)` on conflict — last non-null write wins, NVD data is never overwritten.

## LVE Identity

Every vulnerability gets a stable `LVDB-00000001` identifier generated by PostgreSQL sequence on first insert.

Lookup uses **CVE-prefixed aliases only** to prevent unrelated LVEs from merging through a shared advisory ID (e.g. two CVEs both covered by the same ADV).

```
incoming aliases: ["CVE-2024-1234", "RHSA-2024:001"]
  → match on CVE-2024-1234 in existing lve.aliases
  → if found: UPDATE + merge aliases
  → if not:   INSERT → assigns LVDB-00000042
```

## Notices

When a record cannot be fully mapped, a row is written to `notices` instead of silently dropping data:

```json
{
  "type": "missing_purl",
  "source": "microsoft",
  "message": "No PURL for product: Azure Site Recovery VMWare to Azure",
  "metadata": { "product_id": "...", "advisory_id": "2024-Apr" }
}
```

Queryable via GraphQL so unmapped data is surfaced rather than silently dropped.
